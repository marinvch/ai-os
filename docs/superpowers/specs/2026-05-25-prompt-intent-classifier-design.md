# Prompt Intent Classifier — Design Spec

> **Date:** 2026-05-25
> **Author:** GitHub Copilot (autopilot)
> **Status:** Draft — Awaiting User Review
> **Scope:** Automatic intent detection when a user types a prompt in a repo with ai-os
> installed. Classifies the intent, auto-selects the relevant skill, and asks one
> clarifying question before acting.

---

## Executive Summary

When a developer types a prompt like "add user authentication" or "update the schema to
add bookings", ai-os should proactively:

1. **Classify the intent** — is this a spec-driven feature, a local refactor, a bug fix,
   a DB migration, a dependency upgrade?
2. **Surface the right skill** — route to brainstorming, systematic-debugging, test-driven-
   development, or prisma-database-setup automatically.
3. **Ask one clarifying question** — "Is this a planned spec-driven change or a quick local
   edit?" — to choose the right workflow before any code is touched.

This is a **two-layer system**:

| Layer | Mechanism | Works without | Notes |
|---|---|---|---|
| Static | `copilot-instructions.md` Intent Protocol section | repo-index | Ships in every ai-os install |
| Dynamic | `detect_intent` MCP tool | LSP / embeddings | Reads repo-index for codebase-aware context |

The static layer is the foundation — it works on day one. The dynamic layer enriches it
when the Repository Intelligence Index (RII) is available.

**Out of scope:**
- LLM-based intent classification (external API dependency)
- Real-time keypress monitoring (not possible in Copilot chat)
- Automatic code execution without user confirmation

---

## Problem Statement

Today a developer using ai-os must:
1. Know which skill to invoke (brainstorming? test-driven-development? systematic-debugging?)
2. Know when to write a spec vs just edit code
3. Remember to call `get_session_context` at session start

This is a **workflow burden**. The skill routing table in `copilot-instructions.md` helps
with (1) but only if the developer already knows skills exist. There is no proactive
classification, no "you're about to do something that needs a spec — do you want to use
the brainstorming flow?" moment.

The Prompt Intent Classifier eliminates this burden by making the right workflow the path
of least resistance.

## Design

### Layer 1 — Static Intent Protocol (instructions-based)

A new section injected into every generated `copilot-instructions.md`:

```markdown
## Intent Classification Protocol

Before responding to any implementation, feature, or change request, silently classify
the intent using the table below. If the intent matches a WORKFLOW-FORK pattern, ask
the clarifying question shown — one question only — before proceeding.

| Pattern keywords | Detected intent | Clarifying question |
|---|---|---|
| spec, design, requirement, feature, implement, build | New feature | "Is this a planned feature that needs a spec, or a quick local addition?" |
| fix, bug, error, crash, broken, failing, exception | Bug fix | "Is this a known tracked bug or a quick local fix?" |
| refactor, cleanup, reorganize, extract, rename | Refactor | "Is this a systematic refactor (needs a plan) or a focused local improvement?" |
| schema, table, migration, CREATE TABLE, ALTER TABLE, column | DB change | "Is this a spec-driven schema change or a one-time local edit?" |
| upgrade, update, bump, migrate, version | Dependency change | "Is this a planned upgrade or a quick patch?" |
| test, spec, coverage, failing test | Testing | "Are you adding new tests for a new feature or fixing existing tests?" |

**WORKFLOW-FORK rule**: If you ask the clarifying question and the answer indicates a
spec-driven or planned workflow, invoke the `brainstorming` or `writing-plans` skill
before writing any code. If the answer indicates a local quick edit, proceed directly.

**Skip the question when**:
- The user explicitly says "quick fix", "just", "only", "don't need a spec"
- The change is ≤ 5 lines (comment, rename, obvious typo)
- The user is already in a named skill or plan flow
```

This section is generated by `src/generators/instructions.ts` when building
`copilot-instructions.md`. It is **append-only** (protected block — not overwritten on
`--refresh-existing`).

### Layer 2 — `detect_intent` MCP Tool (dynamic, RII-powered)

When the repo-index exists, the `detect_intent` tool provides codebase-aware intent
classification that goes beyond keyword matching.

```typescript
// Input schema
{
  prompt: string;             // The user's raw message text
  activeFile?: string;        // Currently open file path (if IDE provides it)
}

// Output schema
{
  intentType: IntentType;
  confidence: 'high' | 'medium' | 'low';
  affectedDomain: string[];   // e.g. ['auth', 'database', 'api']
  suggestedSkill: string | null;
  clarifyingQuestion: string | null;
  reasoning: string;          // one sentence — shown to user as context
}

type IntentType =
  | 'new-feature'       // spec-driven work needed
  | 'bug-fix'           // debugging flow needed
  | 'refactor'          // plan needed
  | 'db-change'         // migration flow needed
  | 'test-addition'     // TDD flow needed
  | 'dependency-update' // upgrade flow needed
  | 'docs-update'       // documentation only
  | 'config-change'     // configuration only
  | 'quick-edit';       // local, no workflow needed
```

**How it uses the repo-index:**
1. Extract keywords from the prompt
2. Search `SymbolEntry` + `FileEntry` records for matching names/tags
3. Identify the **domain** (auth, database, API, etc.) from matching entries
4. Cross-reference with `SpecEntry` records — if a matching spec exists, confidence
   goes to `high` and the intent is tagged as spec-driven
5. If no spec found for a "new feature" prompt, `clarifyingQuestion` asks whether
   to create one

**Example:**

User types: *"add login rate limiting"*

Without index → static layer matches `implement` → asks: "Is this a planned feature or quick local addition?"

With index → `detect_intent` finds:
- `src/auth/login.ts` with tags `['auth', 'security']`
- No spec covering rate-limiting
- Returns: `{ intentType: 'new-feature', affectedDomain: ['auth', 'security'], suggestedSkill: 'brainstorming', clarifyingQuestion: "I found your auth module (src/auth/login.ts) but no spec for rate limiting. Should I start a spec-driven design, or add it directly?" }`

The codebase-aware question is **far more useful** than the generic one.

### The Intent Protocol in Generated Instructions

The static Intent Protocol section is a new template block in `src/templates/base-instructions.md`.
It renders into the final `copilot-instructions.md` and is protected from overwrite (added
to the protected blocks list).

Structure within `copilot-instructions.md`:

```
## Session Restart Protocol      ← existing
## Intent Classification Protocol ← NEW (protected)
## Skill Routing                  ← existing (auto-generated)
## MCP Tools Available            ← existing
```

The `detect_intent` tool is also listed in the MCP tools section so Copilot knows to call
it proactively.

---

### The Skill Auto-Selection Enhancement

Currently the skill routing table in `copilot-instructions.md` lists skills and their
trigger descriptions. With intent detection, the instructions additionally say:

```markdown
### Auto-Selection Rule
When `detect_intent` returns a `suggestedSkill`, invoke that skill BEFORE generating
any implementation code. If the user has already explicitly invoked a skill, skip
`detect_intent`.
```

This closes the gap where a developer types "implement OAuth login" and Copilot jumps
straight into code without going through the brainstorming → spec → plan flow.

## Relationship to Repository Intelligence Index

| RII provides | Intent Classifier uses it for |
|---|---|
| `FileEntry.tags` | Match prompt keywords to codebase domains |
| `SymbolEntry.name + file` | Find which file a prompt is likely about |
| `SpecEntry.specId` | Detect if the feature is already spec'd |
| `MetaEntry.primaryFramework` | Tailor clarifying question to the stack |

The intent classifier is a **consumer** of the RII. It should be implemented after Phase 1
of the RII is complete (or in parallel, with the graceful-degradation fallback active).

---

## Acceptance Criteria

1. Every ai-os-generated `copilot-instructions.md` contains the Intent Classification
   Protocol section after a `--refresh-existing` run
2. The section is a protected block — not overwritten on subsequent refreshes
3. `detect_intent({ prompt: "add login rate limiting" })` returns `intentType: 'new-feature'`
   and a non-null `clarifyingQuestion`
4. `detect_intent({ prompt: "fix typo in README" })` returns `intentType: 'quick-edit'`
   and `clarifyingQuestion: null`
5. When `repo-index.jsonl` is absent, `detect_intent` falls back to keyword classification
   without error
6. All unit tests pass; coverage thresholds maintained
