## Feature 1: Structured Error Reporting (#186)

### Problem

`dispatch.ts` and all action files use ad-hoc `console.log('❌ ...')` + `process.exit(1)`. There is no structured error type, no actionable fix suggestion, and no machine-readable exit code. When an error occurs inside a generator, the only output is a generic node stack trace.

### Design

**`src/errors.ts`** — central error module:

```typescript
export type AiOsErrorCode =
  | 'MISSING_CONFIG'
  | 'INVALID_CONFIG'
  | 'WRITE_FAILED'
  | 'SCAN_FAILED'
  | 'TEMPLATE_NOT_FOUND'
  | 'MCP_RUNTIME_MISSING'
  | 'BUNDLE_CORRUPTED'
  | 'UNKNOWN';

export class AiOsError extends Error {
  constructor(
    public readonly code: AiOsErrorCode,
    message: string,
    public readonly fix?: string,
    public readonly details?: unknown,
  ) {
    super(message);
    this.name = 'AiOsError';
  }
}

export function formatError(err: AiOsError): string {
  let out = `\n  ❌ ${err.message}`;
  if (err.fix) out += `\n     Fix: ${err.fix}`;
  if (err.code !== 'UNKNOWN') out += `\n     Code: ${err.code}`;
  return out;
}
```

**`src/generate.ts`** (dispatch.ts main() catch block):

```typescript
} catch (err) {
  if (err instanceof AiOsError) {
    console.error(formatError(err));
    process.exit(err.code === 'UNKNOWN' ? 1 : 2);  // 2 = known recoverable error
  }
  console.error('  ❌ Unexpected error:', err instanceof Error ? err.message : String(err));
  process.exit(1);
}
```

**Exit codes:**
- `0` — success
- `1` — unexpected/unhandled error
- `2` — known AiOsError (with actionable fix message)
- `3` — hygiene / drift check failed (already used in those actions)

**Adoption:** Replace `throw new Error('...')` with `throw new AiOsError(code, message, fix)` in the 5 highest-impact callsites:
1. `generators/utils.ts` — `writeFileAtomic` on EACCES/EROFS
2. `analyze.ts` — package.json parse error
3. `generators/mcp.ts` — missing bundle path
4. `updater.ts` — bundle corrupted check
5. `actions/apply.ts` — top-level catch

### Acceptance Criteria
- `AiOsError` class and `AiOsErrorCode` union exported from `src/errors.ts`
- `formatError()` produces actionable output with fix hint
- `dispatch.ts` main() catch uses structured error formatting
- 5 callsites updated to throw `AiOsError` with correct code + fix message
- Unit tests: `src/tests/errors.test.ts` — 8 tests covering all error codes + formatError output

## Feature 2: Generation Summary Output (#187)

### Problem

After `--refresh-existing`, the CLI prints individual file writes but gives no summary of total impact: how many files changed vs. unchanged, what was trimmed for size, and what the key changes were. Users have no feedback loop for understanding what AI OS changed in their repo.

### Design

**`src/generators/utils.ts`** — extend the `writeIfChanged` tracker:

The file already tracks write hashes via `setPrevHashes`/`getNewHashes`. Extend this to also track:
- Files written (changed)
- Files skipped (unchanged — already written correctly)
- Files trimmed by `enforceSizeCap` (with original vs. trimmed size)

**`src/actions/summary.ts`** — new module:

```typescript
export interface GenerationSummary {
  totalFiles: number;
  changedFiles: string[];
  unchangedFiles: string[];
  trimmedFiles: Array<{ path: string; originalBytes: number; trimmedBytes: number }>;
  durationMs: number;
}

export function formatGenerationSummary(summary: GenerationSummary): string;
export function buildGenerationSummary(startMs: number): GenerationSummary;
```

**Integration in `actions/apply.ts`**:
- Record `startMs = Date.now()` at top of `runApply()`
- After all generators run, call `buildGenerationSummary(startMs)` and print formatted output
- In `--json` mode, emit summary as `{ type: 'generation_summary', ... }` on stdout

**Console output format:**

```
  ── Generation Summary ─────────────────────────
  ✅  12 files written  (4 changed, 8 up-to-date)
  ✂   2 files trimmed  (copilot-instructions.md: 9.2KB → 8.0KB)
  ⏱   Completed in 1.4s
```

### Acceptance Criteria
- `GenerationSummary` type and `formatGenerationSummary()` exported from `src/actions/summary.ts`
- `buildGenerationSummary()` reads from `getNewHashes()` and the trim tracker
- `apply.ts` prints summary after generation (not in preview/plan mode)
- JSON mode emits structured summary object
- Unit tests: `src/tests/summary.test.ts` — 6 tests

## Feature 3: Semantic Drift Detection (#174)

### Problem

Current `detectDrift()` in `src/detectors/drift.ts` only checks **file presence** — whether required AI OS files exist and whether config has the right schema keys. It does NOT detect **content divergence**:
- Framework name in `config.json` doesn't match what's in `copilot-instructions.md`
- `conventions.md` references packages that no longer exist in `package.json`
- `stack.md` lists Node 18 but `package.json` engines says Node 22

### Design

**Extend `DriftItem.kind`** to add `'semantic-mismatch'`:

```typescript
export type DriftKind = 'missing' | 'stale' | 'unknown-file' | 'schema-mismatch' | 'semantic-mismatch';
```

**Add `detectSemanticDrift(cwd: string): DriftItem[]`** — called from `detectDrift()`:

Semantic checks (all implemented as lightweight text comparisons, no LLM):

1. **Framework name consistency** — read `config.json` `primaryFramework`, check that `copilot-instructions.md` contains the framework name (case-insensitive). If config says `nextjs` but instructions contain `express`, emit warning.

2. **Package staleness check** — read packages listed in `stack.md` under "Key Dependencies", check that each one still exists in `package.json` (or `requirements.txt`, etc.). Packages in stack.md but not in any lock/manifest = `warning`.

3. **Node version consistency** — if `stack.md` mentions a Node version (regex: `Node[. ](\d+)`), compare to `package.json` `engines.node` or `.nvmrc`. Mismatch = `info`.

4. **Agent count consistency** — count `.github/agents/*.agent.md` files, compare to `agents.json` `agents.length`. Mismatch = `warning` (agents.json is stale).

5. **Snapshot age** — already partially implemented via `SNAPSHOT_MAX_AGE_DAYS`. Move this check to `detectSemanticDrift` and give it a `'semantic-mismatch'` kind.

**`formatDriftReport()`** — add a "Semantic Issues" section when `kind === 'semantic-mismatch'`.

### Acceptance Criteria
- `DriftItem.kind` extended to include `'semantic-mismatch'`
- `detectSemanticDrift(cwd)` implements all 5 checks above
- `detectDrift()` calls `detectSemanticDrift()` and merges results
- `formatDriftReport()` renders semantic issues in a distinct section
- `--check-drift` CLI flag surfaces semantic issues without changing exit code contract
- Unit tests: `src/tests/drift.test.ts` — 8+ new tests covering each semantic check

## Feature 4: MCP Prompts / Slash Commands

### Problem

VS Code v1.101 added MCP prompt support: prompts registered by an MCP server become slash commands `/mcp.servername.promptname` in VS Code Chat. AI OS has no prompt definitions, so users cannot invoke common AI OS workflows via slash commands — a major discoverability gap.

### Design

**MCP protocol extension in `src/mcp-server/index.ts`**:

Add a `prompts/list` handler and a `prompts/get` handler alongside the existing `tools/call` handler.

**3 prompts to register:**

| Prompt | Slash Command | Argument | Purpose |
|--------|--------------|----------|---------|
| `session_start` | `/mcp.ai-os.session_start` | none | Load context, memory, conventions in one invocation |
| `pre_commit_check` | `/mcp.ai-os.pre_commit_check` | `files` (optional) | Check staged files against conventions + memory |
| `architecture_review` | `/mcp.ai-os.architecture_review` | `focus` (optional) | Structured architecture analysis workflow |

**Prompt message format** (MCP spec `PromptMessage`):

```typescript
interface Prompt {
  name: string;
  description: string;
  arguments?: Array<{ name: string; description: string; required: boolean }>;
}

interface PromptMessage {
  role: 'user' | 'assistant';
  content: { type: 'text'; text: string };
}
```

**`session_start` prompt text:**

```
Call get_session_context, then get_repo_memory, then get_conventions.
Summarize: current stack, active plan, and any open checkpoints.
Do NOT write any files or run any commands.
```

**`pre_commit_check` prompt text (with optional `files` arg):**

```
You are performing a pre-commit code review.
{{files ? `Files to review: ${files}` : 'Review all staged changes.'}}
1. Call get_conventions to load this repo's coding rules.
2. Call get_repo_memory to check for relevant constraints.
3. For each changed file: check naming, imports, error handling against conventions.
4. Report issues as: [SEVERITY] file:line — description — fix.
Do NOT modify files.
```

**`architecture_review` prompt text:**

```
You are performing an architecture review.
Focus area: {{focus || 'full system'}}
1. Call get_project_structure to understand layout.
2. Call get_stack_info to understand the tech stack.
3. Call get_impact_of_change for each core module to understand coupling.
4. Call get_repo_memory for documented architectural decisions.
5. Report: what is well-structured, what has coupling risks, what should be refactored.
```

**JSON-RPC handlers to add in `src/mcp-server/index.ts`**:

```typescript
case 'prompts/list':
  response = { prompts: MCP_PROMPTS };
  break;
case 'prompts/get': {
  const promptName = (req.params as Record<string,unknown>)['name'] as string;
  const args = (req.params as Record<string,unknown>)['arguments'] as Record<string,string> | undefined;
  const prompt = MCP_PROMPTS.find(p => p.name === promptName);
  if (!prompt) {
    response = { error: { code: -32602, message: `Prompt not found: ${promptName}` } };
  } else {
    response = { messages: buildPromptMessages(promptName, args) };
  }
  break;
}
```

### Acceptance Criteria
- `MCP_PROMPTS` array defined in `src/mcp-server/index.ts` with 3 prompts
- `prompts/list` returns all 3 prompts with correct schema
- `prompts/get` returns `PromptMessage[]` for each named prompt
- `prompts/get` with unknown name returns error `-32602`
- `prompts/get` with `session_start` arg substitution works
- Unit tests in `src/tests/mcp-prompts.test.ts` — 6 tests
