# Prompt Booster — Design Spec

> **Date:** 2026-05-25
> **Author:** GitHub Copilot (autopilot)
> **Status:** Draft — Awaiting User Review
> **Scope:** Automatic prompt refinement pipeline. When a user cannot write a precise
> prompt, ai-os asks 1–3 targeted clarifying questions, synthesizes an optimized prompt,
> shows it for confirmation, then hands off to the Intent Classifier and the right skill.

---

## Executive Summary

A developer types: *"add the booking stuff"*

Without a Prompt Booster, Copilot guesses. With it:

1. **Vagueness detected** — short, no target component, no concrete action verb
2. **Question 1:** "What kind of booking — creating new bookings, displaying them, or cancellations?"
3. **Question 2:** "Which layer — database schema, REST API, or frontend form?"
4. **Synthesized prompt shown:**
   > "Add a REST API endpoint for creating new apartment bookings, connecting to the
   > existing `apartments` table. Include date-range validation and return a booking ID."
5. **Confirmation:** "Is this what you meant? (Yes / adjust)"
6. → Intent Classifier → `brainstorming` skill → spec → plan → code

The user typed 5 words and got a precise, actionable prompt — **without needing to know
how to describe the work precisely**.

**Three-spec pipeline for ai-os intelligent routing:**

```
Prompt Booster → Intent Classifier → Skill Selection → Action
    (this spec)      (#2 spec)          (existing)
```

**Out of scope:**
- LLM rewriting without questions (hallucination risk without user grounding)
- Voice input transcription
- Auto-execution without user confirmation step

---

## Problem Statement

Not every developer can describe what they want precisely. Common failure modes:

| Input | Problem | Copilot result without booster |
|---|---|---|
| "add the booking stuff" | Too vague — no layer, no action | Guesses, likely wrong |
| "fix it" | No subject — what is "it"? | Asks "fix what?" (wasteful back-and-forth) |
| "make it faster" | No target component, no metric | Random optimization attempts |
| "the auth is broken" | No error details, no reproduce steps | Generic debugging advice |
| "add schema for DB" | No language/ORM/table specified | Wrong migration dialect |

These are not skill problems — the developer knows what they want, they just can't
articulate it precisely. The Prompt Booster bridges that gap proactively.

## Design

### Two-Layer Architecture (mirrors Intent Classifier)

| Layer | Mechanism | Requires |
|---|---|---|
| Static | `copilot-instructions.md` Prompt Clarification Protocol | Nothing (Day 1) |
| Dynamic | `boost_prompt` MCP tool | `repo-index.jsonl` (RII Phase 1) |

---

### Vagueness Score

Both layers use the same 5-point scoring model to decide whether to trigger:

| Signal | Points |
|---|---|
| Prompt < 10 words | +2 |
| No concrete action verb (create, update, delete, fix, add, remove, migrate...) | +1 |
| No component reference (file path, function name, API route, table name...) | +1 |
| No domain keyword (auth, database, API, UI, schema, test...) | +1 |

**Threshold:** Score ≥ 3 triggers the booster. Score < 3 → pass through to Intent Classifier directly.

**Explicit bypass keywords** (never trigger booster):
- "just", "quickly", "only", "fix typo", "rename"
- Prompt starts with a file path or function name
- User is already in a named skill/plan flow

---

### Layer 1 — Static Prompt Clarification Protocol

New protected section injected into every generated `copilot-instructions.md`:

```markdown
## Prompt Clarification Protocol

**Trigger**: If a message scores ≥ 3 on the vagueness scale (see below), activate the
Prompt Booster before doing anything else.

**Vagueness scale:**
- < 10 words: +2 points
- No action verb (create/update/delete/fix/add/migrate): +1 point
- No component mentioned (file, table, endpoint, function, screen): +1 point
- No domain mentioned (auth, database, API, UI, testing, schema): +1 point

**Booster flow (max 3 questions):**
1. Ask the WHAT question if missing: "What specifically should happen?"
2. Ask the WHERE question if missing: "Which part of the codebase — frontend, API, database?"
3. Ask the HOW question if missing: "Should this be a new addition, a change to existing
   code, or a schema migration?"

**After ≤ 3 answers:**
- Synthesize into one precise, actionable prompt
- Show it: "Here's what I understood: [optimized prompt]. Is this right?"
- Wait for confirmation. On "yes" → hand off to Intent Classification Protocol.
  On adjustment → incorporate and confirm again (one round only).

**Skip the booster when:**
- Prompt includes a file path, function name, or line number
- User says "just", "quickly", "fix typo", "rename X to Y"
- Already inside a skill, plan, or brainstorming flow
```

This section is generated by `src/generators/instructions.ts` and is a **protected block**.
It appears between the Session Restart Protocol and the Intent Classification Protocol.

### Layer 2 — `boost_prompt` MCP Tool (dynamic, RII-powered)

When the repo-index is available, the tool replaces the static questions with
**codebase-aware questions** that use actual file names, table names, and domain tags.

```typescript
// Input schema
{
  prompt: string;         // The user's raw vague message
  activeFile?: string;    // Currently open file path (if known)
}

// Output schema
{
  vaguenessScore: number;         // 0–5
  triggered: boolean;             // true if score >= 3
  questions: ClarifyingQuestion[];
  // Only present after questions are answered:
  optimizedPrompt?: string;
  affectedDomain?: string[];
  suggestedSkill?: string;
  confirmationMessage?: string;   // "Here's what I understood: ..."
}

interface ClarifyingQuestion {
  id: 'what' | 'where' | 'how';
  text: string;              // e.g. "Which booking layer — apartments table, API endpoint?"
  choices?: string[];        // repo-specific suggestions from the index
  required: boolean;
}
```

**How the repo-index enriches questions:**

For "add the booking stuff" + `repo-index.jsonl` containing `apartments` table tags:

| Without index (static) | With index (dynamic) |
|---|---|
| "Which layer — database, API, or frontend?" | "I found `src/db/apartments.ts` and `src/api/routes.ts`. Which layer — add a new `bookings` table, a `/bookings` API endpoint, or both?" |
| "What should happen?" | "Should this create new bookings, list existing ones, or handle cancellations?" (choices from existing CRUD patterns in index) |

The dynamic question uses **actual names from the codebase** — dramatically faster to
answer than generic questions.

---

### Synthesized Prompt Format

After answers are collected, the synthesized prompt follows a fixed template:

```
[ACTION VERB] [COMPONENT] [in/for DOMAIN] [with CONSTRAINTS] [CONTEXT if known]
```

Examples:
- Input: "add booking" + answers: API layer, create new, apartments
- Output: `"Add a POST /bookings API endpoint that creates a new booking for an apartment, validated against the apartments table."`

- Input: "fix auth" + answers: login failing, JWT token, Node backend
- Output: `"Fix the JWT token validation in the Node.js auth middleware — login returns 401 even with valid credentials."`

The synthesized prompt is always shown to the user for confirmation. This is non-negotiable
— the booster never acts on a prompt the user hasn't approved.

---

### Pipeline Integration

The three-spec pipeline runs in strict sequence:

```
User message
  │
  ▼
boost_prompt (or static Clarification Protocol)
  │  vaguenessScore < 3: pass through
  │  vaguenessScore ≥ 3: ask questions → synthesize → confirm
  │
  ▼ optimized (or original) prompt
  │
detect_intent (or static Intent Protocol)
  │  classifies intent type + suggests skill
  │
  ▼
skill invocation (brainstorming / writing-plans / TDD / etc.)
  │
  ▼
implementation
```

**Both tools are called in the copilot-instructions.md rule:**
```markdown
1. ALWAYS call boost_prompt first if vagueness score ≥ 3
2. THEN call detect_intent on the optimized prompt
3. THEN invoke the suggested skill
```

When neither tool is available (MCP server not running), the static protocol sections
handle both steps using Copilot's built-in reasoning.

## Implementation Order

Because Layer 1 (static) has no dependencies, it can ship first:

| Phase | What | Depends on |
|---|---|---|
| **A** | Static Clarification Protocol in `base-instructions.md` | Nothing |
| **B** | `boost_prompt` MCP tool (generic questions, no index) | Nothing |
| **C** | `boost_prompt` + repo-index integration (codebase-aware questions) | RII Phase 1 |
| **D** | Full pipeline integration: booster → intent → skill | Intent Classifier spec |

Phase A ships with the next `--refresh-existing` run. Phases B–D require code changes.

---

## Acceptance Criteria

1. After `--refresh-existing`, `copilot-instructions.md` contains a "Prompt Clarification
   Protocol" section as a protected block (not overwritten on next refresh)
2. `boost_prompt({ prompt: "add the booking stuff" })` returns `triggered: true`,
   `vaguenessScore >= 3`, and exactly 2–3 `questions`
3. `boost_prompt({ prompt: "fix the null check on line 42 of src/auth.ts" })` returns
   `triggered: false`, `vaguenessScore < 3`
4. With `repo-index.jsonl` present, questions reference actual file names / table names
5. Without `repo-index.jsonl`, tool falls back to generic questions without error
6. `optimizedPrompt` follows the `[ACTION] [COMPONENT] [DOMAIN] [CONSTRAINTS]` template
7. All new unit tests pass; coverage thresholds maintained
